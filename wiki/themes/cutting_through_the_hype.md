The AI engineering landscape is thick with terminology that sounds more complex than the engineering it describes. RAG. MCP. A2A. Agent harnesses. Skills. Context engineering. Each term arrives with a wave of blog posts, vendor pitches, and conference talks that make it seem like an entirely new discipline. And then Stefan Krawczyk stands up in [[ws_8_finetuning_and_production_ai]] and says, flatly: "It's all just an API call."

Stefan's [[the_api_call_mental_model]] is the single most powerful hype-deflation tool in the course. Every LLM interaction reduces to a call with specific components: a system prompt, a sequence of messages, tool schemas, and structured output definitions. Every buzzword maps to a specific part of this call. RAG populates the context. MCP populates the tool schemas. Guardrails validate context on the way in and responses on the way out. Skills are progressive disclosure of context -- the agent reads files to add information as needed rather than loading everything upfront. Once you understand these mappings, concepts that seemed opaque become obvious patterns. Natalia Rodnova's experience confirms this: after understanding the basics, RAG and MCP stopped seeming complex and started seeming like "obvious patterns."

The framework's diagnostic power extends beyond pedagogy. When someone describes a product and you cannot map its functionality to specific parts of an API call, Stefan argues, either you do not understand it or it is doing less than advertised. This is a particularly sharp tool for evaluating the [[framework_selection]] landscape, where vendors have strong incentives to make things sound complex. Hugo's demonstration of Llama Index's 5-line quickstart -- which hides the model, the prompt, and all retrieval internals -- is the canonical example of a framework that trades understanding for convenience. The convenience is real, but so is the debt it creates when the system misbehaves and you have no visibility into what went wrong.

The terminology debate around [[agent_harnesses]] crystallizes the broader tension. Stefan dismisses "harness" as marketing jargon that people use to sell to executives. Hugo argues it usefully encapsulates tools, context management, and state tracking in a single term. Carol Willing cuts through both positions: "Good engineering is good engineering, you can name it My Pink Pony or the trapeze of the month." Through the API call lens, a harness is simply whatever surrounds the call -- the code that assembles the prompt, manages tools, tracks state, and processes the response. Whether you call it a harness, a wrapper, or just "the rest of the code" is a naming choice, not a technical distinction.

The [[agents_vs_workflows]] distinction is perhaps the most consequential place where terminology creates confusion. Hugo identifies three definitions of "agent" depending on who is talking: an AI person sees an LLM calling tools in a loop, an infrastructure person sees a microservice, and a business person sees something that can replace a human. Most production AI applications are workflows -- systems with predefined code paths where the developer controls the sequence -- even when stakeholders call them "agents." Hugo cites Eric Ma's audit at Moderna: everything in production was a workflow, not an agent. William Horton's practical heuristic cuts through the confusion: get people to describe desired inputs and outputs without using hype terms like "agentic." If they cannot describe what they want without buzzwords, they do not yet understand the problem.

The relationship between [[prompt_engineering]] and [[context_engineering]] is another boundary that the API call model clarifies. William Horton questions whether context engineering is really different from prompt engineering -- a challenge that highlights the blurry boundary. The resolution offered across multiple sources is one of automation and scale: prompt engineering is what a human writes; context engineering is building the machine that writes it. Lance Martin's four patterns -- write (offload to scratchpads), select (retrieve relevant context), compress (summarize), and isolate (delegate to sub-agents) -- are all strategies for populating the messages array of the API call. They are not a new paradigm; they are systematic approaches to a problem that grows as systems scale beyond what a human can hand-craft.

[[structured_output]] provides another clarifying example. It sounds like a feature, but through the API call lens it is a load-bearing component of the call anatomy, sitting alongside system prompt, messages, and tool schemas. Structured output enables the gate pattern that distinguishes workflows from free-form agents: the LLM produces a structured classification, and downstream code routes deterministically based on that output. It makes evaluation possible by giving you a schema to validate against. It makes guardrails work by providing parseable safety assessments. Stefan's veterinary transcription DAG -- 60+ LLM calls in a directed acyclic graph -- is debuggable precisely because every intermediate representation between pipeline stages is structured and inspectable.

[[tool_calling_and_function_calling]] is similarly demystified through the API call model. The LLM does not execute tools; it returns text describing which tool to call and with what arguments. The developer parses, executes, and returns results. MCP is a distribution protocol that populates the tool schema section of the call -- standardized, useful for sharing tools externally, but not a new form of AI capability. The consensus from the Builders Club is telling: MCP servers are primarily useful for distributing tools to external users. For personal or team use, plain tool calling is simpler and sufficient.

The most interesting development in this landscape is that models are absorbing the infrastructure that previously surrounded them. Through RLVR -- reinforcement learning with verifiable rewards -- models trained on agent traces are learning to perform harness functions internally. Doug Turnbull's Kirby analogy captures this: models absorb the harnesses around them, causing labs to rip out their infrastructure with each new release. Nicholas Moy advises builders to "rip out the harness, let your model be free." This trend does not invalidate the API call mental model -- if anything, it makes it more important. As capabilities migrate between the model and the surrounding code, the API call remains the stable interface where you can observe exactly what is happening. The buzzwords will keep changing. The call structure will not.
