Every complex AI system, no matter how sophisticated, is made of simple parts. A 60-call veterinary transcription DAG. A healthcare agent that routes patients to five specialized sub-agents. A deep research system that spawns sub-agents dynamically to investigate questions. Each of these begins with the same atomic unit: a single API call with a system prompt, messages, tool schemas, and structured output definitions. The story of building intelligent systems is the story of composing this unit into increasingly powerful arrangements.

The foundation is [[the_api_call_mental_model]], which Stefan Krawczyk develops in [[ws_8_finetuning_and_production_ai]] into a comprehensive mapping of industry terminology to API components. Understanding the call is a prerequisite for understanding everything built on top of it. But the call by itself -- a prompt goes in, text comes out -- has limited utility. The first extension that transforms it into something architecturally useful is [[structured_output]].

Structured output constrains the model's response to conform to a schema. This sounds like a convenience feature. It is not. It is the mechanism that enables typed, validated data pipelines with LLMs as components. When Hugo demonstrates the gate pattern -- structured output combined with if-else logic to route between processing paths -- he is showing the fundamental building block of what the course calls "workflows." The LLM produces a structured classification ("intent: appointment" or "intent: health_question"), and downstream code routes deterministically. This pattern keeps workflows predictable while leveraging LLM intelligence at each step. Stefan's veterinary transcription DAG uses structured output at every stage, making each of its 60+ calls independently testable and debuggable. The parallelization trick -- running multiple structured extractions in parallel and aggregating results -- exploits structured output for ensemble-style accuracy improvements. Carol Willing's practice of having three different LLMs audit code files in parallel, connected to Karpathy's LLM Council concept, depends entirely on shared schemas that make programmatic comparison possible.

[[tool_calling_and_function_calling]] is the second transformative extension. The mechanics are deceptively simple: the developer describes available tools, the LLM returns a function name and arguments as text, the developer executes the function and sends results back. But this mechanism is what allows LLMs to interact with the world -- databases, APIs, search engines, code execution environments. Hugo reveals a striking cross-provider inconsistency: the same tool description works on GPT-5 but fails on Gemini 2.5 Flash, requiring iteration per provider. William Horton's pragmatic response -- a dedicated "Day of Week for Date" tool because LLMs are unreliable at calendar reasoning -- exemplifies the principle: externalize computations the model cannot do reliably into deterministic tools rather than fighting the weights.

With structured output providing type safety and tool calling providing external capability, [[retrieval_augmented_generation]] adds external knowledge. Through the API call lens, RAG simply populates the context -- it adds retrieved information to the messages array. But the retrieval stack itself ranges from BM25 keyword search (Hugo's recommended baseline) through embedding-based vector retrieval to sophisticated agentic retrieval where the LLM orchestrates multi-step search strategies. The "Gather and Glean" framework provides the guiding principle: first prioritize recall to gather all potentially relevant content, then re-rank with precision. What classic single-pass RAG fails at -- summarization queries, multi-document comparison, multi-step reasoning -- motivates the move toward agentic retrieval, where the LLM operates in a loop with search tools.

[[context_engineering]] generalizes retrieval into a broader discipline. Lance Martin's four patterns -- write (offload to scratchpads), select (retrieve relevant context), compress (summarize to fit context limits), and isolate (delegate to sub-agents with their own focused context) -- are strategies for managing the most fundamental constraint in LLM systems: the context window is finite, and what goes into it determines what comes out. The "skills" pattern illustrates this concretely: rather than loading all instructions upfront, the agent reads relevant skill files only when it encounters a task that requires them. Stefan calls this "just a name for smart context engineering." Ravin Kumar's account of Notebook LM progressively simplifying its retrieval infrastructure as Gemini's context window grew to one million tokens shows context engineering adapting to changing constraints -- the strategies stay the same, but the specific trade-offs shift as model capabilities evolve.

These building blocks compose into the two major architectural patterns. [[agents_vs_workflows]] is not a binary choice but a spectrum. Plain LLMs handle single-turn text generation. Augmented LLMs add tools and retrieval. Workflows chain multiple LLM calls in predefined patterns with structured output gates controlling the flow. Agents give the LLM control over multi-step processes with dynamic tool selection. The course's central thesis is that most production systems sit closer to the workflow end: predictable paths, bounded costs, testable behavior. Hugo's "Stop Building AI Agents" essay captured this -- not opposing agents per se, but opposing agents deployed without the evaluation and observability infrastructure that workflows make easier to implement.

At the furthest end of the complexity spectrum, [[multi_agent_architecture]] decomposes problems across multiple specialized agents coordinating through a lead orchestrator. William Horton's Maven Assistant routes to sub-agents for appointments, provider search, health questions, and FAQ retrieval. Ivan Leo's deep research agent spawns sub-agents dynamically, each with iteration limits and isolated context. Context isolation -- the "isolate" pattern from Lance Martin's framework -- is what makes this tractable. Without it, the orchestrator's context window would fill with raw details from every sub-agent's work.

[[model_selection_and_tradeoffs]] operates across every level of this architecture. The Pareto frontier runs from tiny models (Gemma 270M on a Raspberry Pi) through Flash-tier workhorses (Gemini Flash for latency-sensitive production) to frontier reasoning models (capable but too slow and expensive for most real-time use). The emergent pattern -- smart models for planning, cheap models for execution -- appears independently in William Horton's lead-agent-plus-sub-agents architecture, Ivan Leo's deep research orchestration, and Eric Ma's explicit development strategy.

The most provocative extension of these building blocks is [[coding_agents_as_general_purpose_agents]]. A coding agent built in 131 lines of Python, with just four tools -- read, write, edit, and bash -- can organize files, write scripts, create new tools, and hot-reload them at runtime. Ivan Leo's question captures the philosophical implication: "Isn't every agent a coding agent?" The logic is that any useful general-purpose agent eventually needs to execute arbitrary operations, and code execution is the universal interface. Hugo's demonstration of an agent writing its own Whisper transcription tool -- encountering a voice message it cannot process, creating the tool, then using it -- shows self-extension emerging naturally from simple building blocks.

The journey from a single API call to a self-extending multi-agent system is a story of composition, not complexity. Each layer adds capability by combining the pieces below it, and each piece maps back to the atomic unit that Stefan described. Understanding that unit -- what goes in, what comes out, which knobs exist -- remains the foundation. Everything else is arrangement.
